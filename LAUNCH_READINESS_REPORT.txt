1. EXECUTIVE SUMMARY
================================================================================

Rana Market adalah super-app UMKM yang sangat ambisius dengan ekosistem
lengkap (POS + Marketplace + Ride-Hailing + B2B + PPOB). Dari sisi fitur,
app ini sudah 80-85% selesai. Namun dari sisi production-readiness, ada
beberapa celah keamanan dan stabilitas yang HARUS ditutup sebelum launch.

Rekomendasi dibagi menjadi 3 fase:

  FASE              DURASI         DESKRIPSI
  --------------------------------------------------------------------------
  BLOCKER           1 minggu       Wajib selesai. Gagal = risiko data breach
  PENTING           1-2 minggu     Sebaiknya selesai sebelum launch publik
  POST-LAUNCH       Setelah launch Optimisasi sambil jalan

----------------------------------------------------------------------
2.1 KEAMANAN JWT - KRITIS
----------------------------------------------------------------------

MASALAH:
  JWT secret key memiliki fallback hardcoded 'super_secret_key_change_in_prod'
  yang tersebar di 6 FILE:

    - server/src/middleware/auth.js
    - server/src/socket.js
    - server/src/services/digiflazzService.js
    - server/src/controllers/authController.js
    - server/src/controllers/adminController.js (2 tempat)

  BAHAYA: Siapapun yang melihat source code bisa membuat token JWT palsu
  dan mengakses SEMUA data merchant.

SOLUSI:
  Buat file: server/src/config/secrets.js

    const JWT_SECRET = process.env.JWT_SECRET;
    if (!JWT_SECRET) {
        console.error('FATAL: JWT_SECRET not set in environment!');
        process.exit(1); // Server TIDAK BOLEH jalan tanpa secret
    }
    module.exports = { JWT_SECRET };

ESTIMASI: 30 menit

----------------------------------------------------------------------
2.2 PRISMA CLIENT SINGLETON - KRITIS
----------------------------------------------------------------------

MASALAH:
  Ditemukan 38 INSTANCE 'new PrismaClient()' tersebar di semua file.
  Setiap file controller membuat koneksi database sendiri.

  BAHAYA: Di production dengan PM2 cluster mode, ini bisa menghabiskan
  semua koneksi PostgreSQL (default: 100).
  Dengan 38 instance x 4 cluster = 152 koneksi - server akan CRASH.

SOLUSI:
  Buat file: server/src/config/database.js

    const { PrismaClient } = require('@prisma/client');
    let prisma;
    if (process.env.NODE_ENV === 'production') {
        prisma = new PrismaClient({ log: ['warn', 'error'] });
    } else {
        if (!global.__prisma) {
            global.__prisma = new PrismaClient({ log: ['query', 'warn', 'error'] });
        }
        prisma = global.__prisma;
    }
    module.exports = prisma;

  Lalu di SEMUA controller/middleware, ganti:
    HAPUS:  const { PrismaClient } = require('@prisma/client');
    HAPUS:  const prisma = new PrismaClient();
    GANTI:  const prisma = require('../config/database');

ESTIMASI: 1-2 jam (mekanis, tapi banyak file)

----------------------------------------------------------------------
2.3 INPUT VALIDATION - KRITIS
----------------------------------------------------------------------

MASALAH:
  Tidak ada validasi input di sisi server. Semua endpoint langsung
  memproses req.body tanpa sanitasi.

  Contoh berbahaya di authController.js:
    // Line 143-158: req.body langsung di-destructure tanpa validasi
    const { businessName, ownerName, email, password, ... } = req.body;

  BAHAYA: Tanpa validasi, attacker bisa mengirim payload berbahaya
  (SQL injection via Prisma, oversized strings, wrong types) yang
  menyebabkan crash atau data corruption.

SOLUSI:
  Install: npm install zod

  Buat file: server/src/validators/auth.js
    const { z } = require('zod');
    const registerSchema = z.object({
        businessName: z.string().min(2).max(100),
        email: z.string().email().max(255),
        password: z.string().min(6).max(128),
        waNumber: z.string().min(10).max(15),
    });

  PRIORITAS ENDPOINT: auth/register, auth/login, transactions, wallet, orders

ESTIMASI: 1-2 hari (fokus di endpoint kritis dulu)

3.1 STRUCTURED LOGGING
----------------------------------------------------------------------

MASALAH: Server hanya menggunakan console.log dan Morgan.
         Di production, log tidak terstruktur dan sulit di-debug.

SOLUSI:
  Install: npm install pino pino-pretty

  Buat file: server/src/config/logger.js
    const pino = require('pino');
    const logger = pino({
        level: process.env.LOG_LEVEL || 'info',
        transport: process.env.NODE_ENV !== 'production'
            ? { target: 'pino-pretty' }
            : undefined,
    });
    module.exports = logger;

ESTIMASI: 1 hari

----------------------------------------------------------------------
3.2 RATE LIMITING PER-ROUTE
----------------------------------------------------------------------

MASALAH: Rate limit = 3000 req/15min untuk SEMUA endpoint.
         Endpoint sensitif (login) dan ringan (get products)
         diperlakukan sama.

SOLUSI:
  // Strict untuk auth
  const authLimiter = rateLimit({ windowMs: 15*60*1000, max: 20 });
  app.use('/api/auth/login', authLimiter);
  app.use('/api/auth/register', authLimiter);

  // Moderate untuk write
  const writeLimiter = rateLimit({ windowMs: 15*60*1000, max: 300 });
  app.use('/api/transactions', writeLimiter);
  app.use('/api/wallet', writeLimiter);

  // General (lebih longgar)
  const generalLimiter = rateLimit({ windowMs: 15*60*1000, max: 1500 });
  app.use('/api/', generalLimiter);

ESTIMASI: 2 jam

----------------------------------------------------------------------
3.3 DATABASE INDEXING
----------------------------------------------------------------------

MASALAH: Schema Prisma tidak memiliki index custom.
         Dengan data bertumbuh, query akan melambat.

SOLUSI: Tambahkan index di schema.prisma:

  model Transaction {
      @@index([tenantId, createdAt])
      @@index([storeId, createdAt])
      @@index([paymentStatus])
  }

  model Product {
      @@index([tenantId, isActive])
      @@index([storeId])
      @@index([barcode])
  }

  model CashflowLog {
      @@index([tenantId, storeId, occurredAt])
  }

  model PpobTransaction {
      @@index([tenantId, createdAt])
      @@index([refId])
  }

ESTIMASI: 1 jam + migration

3.6 PASSWORD POLICY
----------------------------------------------------------------------

MASALAH di authController.js:109:
  isStrongPassword hanya cek panjang > 6 karakter.

SOLUSI:
  const isStrongPassword = (value) => {
      const password = (value || '').toString();
      if (password.length < 8) return false;
      if (!/[A-Z]/.test(password)) return false;  // Min 1 huruf besar
      if (!/[a-z]/.test(password)) return false;  // Min 1 huruf kecil
      if (!/[0-9]/.test(password)) return false;  // Min 1 angka
      return true;
  };
